Section 0

The Prompt Quality Gap

Most developers write prompts the way they'd write a Stack Overflow question. But Claude isn't googling for answers — it's a reasoning engine that needs structure, context, and constraints to produce reliable output.

Prompt Comparison

Both prompts ask Claude to review a PR. But they produce radically different results.

Vague
"Review this PR and tell me if it's good."
Structured
"Review the attached PR diff as a senior backend engineer focusing on security vulnerabilities and database query performance. Flag any N+1 queries, missing input validation, or SQL injection risks. Return your findings as a JSON array with fields: file, line, severity (critical/high/medium/low), issue, and recommended_fix. Skip style comments."
Prediction Challenge
What made the second prompt produce dramatically better output? List as many factors as you can think of — even partial answers count.
The Five Dimensions of Reliable Prompts
  • 1
    Role / Persona

    Who Claude should be. Affects tone, depth, vocabulary, and what knowledge it foregrounds.

  • 2
    Task specificity

    Exactly what to do — not vague direction. "Flag N+1 queries" beats "review the code."

  • 3
    Context

    What Claude needs to know to do the task well: stack, constraints, conventions, background.

  • 4
    Constraints

    What to exclude, what to limit, what matters and what doesn't. "Skip style comments" is a constraint.

  • 5
    Output format

    Structured output (JSON, markdown, numbered list) produces predictable, parseable results every time.

Section 1

The TCEF Pattern

TCEF = Task + Context + Examples + Format. A systematic four-part structure for prompts that produce consistent, structured output every time.

T
Task
What specifically should Claude do? Use precise verbs: "Extract", "Classify", "Generate", "Summarize as", "Convert to".
✗ "Look at this code" ✓ "Identify all functions that mutate state and explain the side effects"
C
Context
What does Claude need to know? Stack, codebase conventions, user background, relevant constraints. Rule: If removing it would change Claude's response, include it.
"Python 3.11 FastAPI. We use Pydantic v2 for all input validation."
E
Examples
Show Claude what good output looks like (few-shot learning). One example beats three paragraphs of description. For classification: show at least one example per class.
Validated: @app.post("/users") def create_user(user: UserCreate): ...
F
Format
Specify the exact output structure: JSON schema, markdown headers, numbered list, prose paragraph. The more specific, the more reliable.
{"endpoint": "/path", "method": "POST", "severity": "high", "fix": "..."}
TCEF Builder — Apply Task

Scenario: You want Claude to review API endpoints in a Python FastAPI codebase and flag any missing input validation. It should return a structured report. Build the four TCEF components below.

T — Task
Verb + what to find + what to produce. Be specific about what "missing validation" means.
C — Context
Language, framework, version. What does "missing validation" mean in your codebase? What tools do you use?
E — Examples
Show one example of a properly validated endpoint vs one that's missing validation. Code snippets work great here.
F — Format
JSON array? Markdown report? What fields does each finding need? endpoint, method, issue, risk, fix?
Your Assembled TCEF Prompt


      

        
Model Answer

        
Task: Review the FastAPI route handlers in the provided code and identify any endpoints missing proper input validation. For each, explain the security risk.

Context: This is a Python 3.11 FastAPI application. "Missing validation" means: no Pydantic model for request body, no type annotations on path/query params, or accepting `Any` type. We use Pydantic v2.

Examples:
Properly validated:
@app.post("/users")
def create_user(user: UserCreate):  # Pydantic model validates input
    ...

Missing validation:
@app.post("/upload")
def upload(data: dict):  # dict accepts anything — no validation
    ...

Format: Return a JSON array. Each item:
{"endpoint": "/path", "method": "POST", "issue": "description", "risk": "security impact", "fix": "recommended fix"}

Lab 02 Callback · CLAUDE.md Context

          
In Lab 02, you learned that CLAUDE.md is Claude's persistent memory for a project — loaded at every session start. That's Strategy 1 below. The other two strategies give you more control over timing and freshness.

        

      


      

        

          

            
S1

            

              
Session-persistent context

              
CLAUDE.md

            

          

          
CLAUDE.md is read at session start — always available throughout the session without any extra effort.

          
CLAUDE.md: "We use PostgreSQL 15 with raw SQL (no ORM). Always use parameterized queries."

          
 Trade-off: Everything in CLAUDE.md uses tokens on every interaction. Keep it focused and stable.

          
 Rule: If it never changes, put it in CLAUDE.md.

        


        

          

            
S2

            

              
Task-time context injection

              
@file references or paste

            

          

          
Include relevant files or data directly in your prompt using @file.py references or pasting content inline.

          
"Review @src/api/routes.py and @src/models/user.py for validation issues"

          
 Trade-off: Uses more tokens per request, but ensures Claude sees exactly what it needs.

          
 Rule: If it's relevant to this specific task, inject it now.

        


        

          

            
S3

            

              
MCP-driven dynamic context

              
Claude pulls via MCP tools at reasoning time

            

          

          
Claude uses MCP tools to fetch live data at reasoning time — fresh context on demand without manual injection.

          
A get_schema MCP tool returns the current DB schema when Claude needs it

          
 Trade-off: Adds latency and cost, but keeps context fresh and automated.

          
 Rule: If it changes between sessions, expose it via MCP.

Section 3

      
Prompt Iteration

      
First-draft prompts rarely produce optimal output. Systematic iteration — not random tweaking — is the skill that separates prompt engineers from prompt guessers.

    


    

      Professional prompt engineers don't write the perfect prompt on the first try. They have a system for knowing what to change and why. Random tweaking is not iteration — it's guessing.
    


    

      
 The Iteration Loop

      

        
Write Prompt

        
→

        
Run It

        
→

        
Identify Failure Mode

        
→

        
Fix Specific Component

        
→

        
Repeat

      

    


    

      
 Six Failure Modes & Targeted Fixes

      

        
          

            Failure Mode
            Symptom
            Fix
          

        
        
          

            Too vague
            Claude asks clarifying questions
            Add Task specificity
          

          

            Wrong persona
            Tone or depth is off
            Add/change Role in Task
          

          

            Missing context
            Claude makes wrong assumptions
            Add Context
          

          

            Wrong format
            Output hard to parse or use
            Add/fix Format
          

          

            No examples
            Output varies wildly
            Add Examples
          

          

            Too broad scope
            Claude includes irrelevant things
            Add Constraints

 Model Progression

        

          

            
v1 — Original (broken)

            
"Summarize this meeting transcript."

          

          

            
v2 — Format fixed

            
"Summarize the key decisions and action items from this engineering team meeting transcript. Format as markdown with two sections: ## Decisions and ## Action Items."

          

          

            
v3 — Full TCEF

            
"You are a technical program manager. Extract key decisions and action items from the attached engineering standup transcript. Context: This is a 15-person backend team; action items should include owner name and deadline if mentioned. Format: ## Decisions (bullet list), ## Action Items (table: Owner | Task | Deadline). Exclude general discussion."

          

        

        

          Notice the progression: format → context → persona + constraints. Each iteration targeted ONE specific failure mode. Systematic, not random.
