Production Patterns

Your agentic workflow works on your laptop. Now it needs to work safely, affordably, and visibly — at organizational scale.

After this lab you will be able to:
  • Configure settings.json permission scoping to enforce least-privilege for any Claude Code agent
  • Apply secrets management patterns that keep credentials out of prompts, logs, and version control
  • Estimate token cost for agentic workflows and identify the highest-leverage optimization levers
  • Design structured observability so any agent session can be replayed and debugged from logs alone

Section 0 · Concept Introduction

The Production Gap

You've built an agentic workflow that works on your laptop. Before shipping it, three questions: What can it touch? What can it spend? What can you see when it fails?

CALLBACK

PRAO Loop — first introduced in Lab 01, Section 1

Perceive → Reason → Act → Observe — the four-phase agentic execution cycle.

Production controls operate on each PRAO phase: Safety constrains Act, Cost constrains Reason depth, Observability captures every phase for audit.

INFO

The Three Production Pillars

1
Safety
Scope what the agent can access and act on. Permissions and sandboxing limit blast radius when things go wrong — and things always go wrong eventually.
2
Cost
Understand and control token spend before it surprises you. Agentic workflows compound token use — one poorly scoped task can read hundreds of files.
3
Observability
Know what happened, when, and why — especially when it fails. Without structured logs, debugging a 2am agent failure is pure guesswork.

The settings.json Permission Model

Permissions live in .claude/settings.json (project level) or ~/.claude/settings.json (global). Project-level settings override global for that project.

{ "permissions": { "allow": [ "Read", // allow all reads (no path restriction) "Write(src/**)", // allow writes only inside src/ "Bash(git *)", // allow any git subcommand "Bash(npm test)" // allow exactly this command ], "deny": [ "Bash(rm *)", // deny all rm commands "Bash(curl *)", // deny outbound HTTP from shell "Write(/etc/**)", // deny writes to system dirs "Write(*.env)" // deny writing to any .env file ] } }
Three levels:
1. ~/.claude/settings.json — global defaults for all projects
2. .claude/settings.json — project-specific; overrides global for this repo
3. Inline approval prompts — runtime override for one-off operations

Challenge CE-08-A: Write Production Permission Models

Real production agents need precise permission models — not "allow everything" but not "block everything" either. Write the complete settings.json permissions block for each scenario below. Every prohibited action needs an explicit deny entry.

SCENARIO 1 CI Test Runner

"A CI agent that reads source code, runs tests, and reports results. Should never write files, commit, or push."

SCENARIO 2 Documentation Agent

"Reads source files and writes to /docs/ only. Can run linters but not tests. Must not touch any .env files."

SCENARIO 3 Production Monitor

"Reads logs and queries a read-only database. Must never write anything, execute shell commands, or access source code."

Validation rules — your submission will be checked for:
  • Valid JSON with both "allow" and "deny" arrays
  • Scenario 1: deny must include Write, Edit, Bash(git commit), Bash(git push)
  • Scenario 2: allow Write must be scoped to docs/ only — not Bash(*)
  • Scenario 3: deny must include Bash and Write — read-only operations only
  • No scenario uses Bash(*) in allow — that's a security red flag
Model Answers:

Scenario 1 — CI Test Runner:

{
  "allow": [
    "Read",
    "Bash(npm test)",
    "Bash(npm run test:*)",
    "Bash(jest *)",
    "Bash(pytest *)"
  ],
  "deny": [
    "Write",
    "Edit",
    "MultiEdit",
    "Bash(git commit*)",
    "Bash(git push*)",
    "Bash(rm *)",
    "Bash(sudo *)"
  ]
}

Scenario 2 — Documentation Agent:

{
  "allow": [
    "Read",
    "Write(docs/*)",
    "Edit(docs/*)",
    "Bash(eslint *)",
    "Bash(prettier --check *)"
  ],
  "deny": [
    "Write(.env*)",
    "Edit(.env*)",
    "Read(.env*)",
    "Bash(npm test*)",
    "Bash(jest*)",
    "Bash(git push*)",
    "Bash(git commit*)"
  ]
}

Scenario 3 — Production Monitor:

{
  "allow": [
    "Read(logs/*)",
    "mcp__database__query_readonly"
  ],
  "deny": [
    "Write",
    "Edit",
    "MultiEdit",
    "Bash",
    "Read(src/*)",
    "Read(.env*)"
  ]
}
Key principle: Deny lists should be specific to what you've identified as dangerous for each scenario — not a generic list. The CI runner doesn't need to deny Read because that's fine; it needs to deny the specific destructive operations (write, commit, push).

Secrets Management

Secrets passed through Claude Code prompts become part of the context window — and potentially part of logs, traces, and outputs. Never inject secrets through prompts.

Three Secrets Anti-Patterns and Their Fixes
Anti-Pattern 1 — Secret in prompt
claude -p "Use API key sk-abc123 to call the OpenAI API"
The key is now in prompt history, logs, and potentially the model's output.
Fix — Secret in environment
export OPENAI_API_KEY=sk-abc123 claude -p "Call the OpenAI API using the key in OPENAI_API_KEY"
Claude Code reads environment variables; the key value never enters the prompt.
Anti-Pattern 2 — Secret hardcoded in CLAUDE.md
API_KEY: sk_live_abc123realkeyhere
CLAUDE.md is committed to version control. Every key becomes repo history forever.
Fix — Reference the env var name, not the value
API keys are in environment variables. See .env.example for names.
Anti-Pattern 3 — Inline secret in MCP config (committed)
{ "env": { "API_KEY": "sk-real-key-here" } }
MCP server configs in the repo expose secrets to anyone with repo access.
Fix — Environment variable substitution
{ "env": { "API_KEY": "${INTERNAL_API_KEY}" } }
The actual value lives in CI secrets or a local .env (gitignored). The config is safe to commit.

Section 3 · Interactive Workshop

Cost Optimization

Token cost is the unit economics of agentic AI. Understanding where tokens go lets you optimize spend without sacrificing capability.

Four Cost Levers
Lever 1 — Context Window Management

Every file you ask Claude to read consumes input tokens. Target reads — don't read entire codebases. Read(src/api/routes.ts) is cheaper than "read the whole project."

Lever 2 — Task Scoping

claude -p for single-shot tasks is cheaper than interactive sessions for scripted workflows. Avoid open-ended sessions when the task is well-defined.

Lever 3 — Model Selection

Use the smallest capable model per task type. Formatting, classification, and simple extraction don't need Sonnet — Haiku-class models handle them at a fraction of the cost.

Lever 4 — Prompt Caching

Stable system prompts and CLAUDE.md content can be cached. On repeated runs, cached content costs ~90% less. A 500-line CLAUDE.md read 100×/day is a significant caching target.

Section 4 · Concept Introduction + Apply

Observability

When an agentic workflow fails in production at 2am, you have one question: what exactly did the agent do? Without observability, you're guessing.

Three Observability Layers
Layer 1 — Structured Logging

Log every tool call with timestamp, tool name, arguments, result, and duration. Use JSON format for queryability — structured logs can be filtered, aggregated, and searched with standard tooling.

Layer 2 — Trace Correlation

Assign a trace_id to each agent session. Every log entry carries the trace ID. Lets you reconstruct a complete session from scattered log lines across multiple files or services.

Layer 3 — Human Audit Trail

For approval-gate workflows, log what was shown to the human and what they approved or rejected. This is the legal and compliance layer — it proves a human was in the loop.

Structured Log Entry — Example

Every tool call produces one structured log entry. The trace_id links all entries for a session.

{ "trace_id": "ag-7f3a9c", "timestamp": "2026-03-15T14:23:01Z", "tool": "Write", "args": { "path": "src/api/routes.ts", "content_length": 847 }, "duration_ms": 234, "status": "success", "session_id": "ses-4b2e1a" }

INSIGHT

Structured logs answer "what happened". Traces answer "in what sequence". Together they let you replay any agent session deterministically — the foundation of incident response for agentic systems.

The diagnostic flow: filter logs by trace_id → sort by timestamp → identify all Write tool calls → cross-reference the modified files with the failing test paths → inspect the args.content_length and adjacent log entries for the specific write that correlates with the first test failure.

Production Deployment Specification

Context: You're deploying an automated PR review agent to production. It will run on every PR in your company's monorepo — 50+ PRs per day, touching 200+ files each. The agent reads diffs, leaves review comments via GitHub API, and never merges or pushes directly.
BOSS CHALLENGE

Write your complete production deployment specification covering all four areas:

  1. settings.json permissions block — what can it read, write, execute?
  2. Secrets handling — what secrets does it need and how are they passed safely?
  3. Cost estimate reasoning — at 50 PRs/day, ~20 files per PR on average, what's your monthly estimate?
  4. Observability design — what do you log, how do you alert on failures, what's your incident response entry?
80 characters to submit
Model Answer
1. Permissions — Read-only, safe git commands only
{ "permissions": { "allow": [ "Read", // read any file (diff inspection) "Bash(git diff *)", // read-only git operations "Bash(git log *)" // read commit history ], "deny": [ "Write", // no file writes — review only "Bash(git push *)", // cannot push anything "Bash(git merge *)", // cannot merge "Bash(rm *)" // no deletions ] } }
2. Secrets Handling
// MCP server config (safe to commit): { "mcpServers": { "github": { "env": { "GITHUB_TOKEN": "${GITHUB_TOKEN}" } } } } // Developer setup: export GITHUB_TOKEN=ghp_... in ~/.zshrc or .env (gitignored) // CI: store in CI secrets vault; inject as environment variable at runtime // NEVER: pass token in -p prompt, CLAUDE.md, or any committed config
3. Cost Estimate
50 PRs/day × 20 files/PR × 2,000 tokens/file = 2,000,000 input tokens/day
Reasoning per PR: ~3,000 tokens × 50 = 150,000 reasoning tokens/day
Output (comments): ~500 tokens × 50 = 25,000 output tokens/day
Input cost: 2.15M × $3/MTok = $6.45/day
Output cost: 25K × $15/MTok = $0.38/day
Total: ~$6.83/day → ~$205/month uncached
With prompt caching on the system prompt: ~$125/month (40% reduction)
4. Observability Design
Log per PR: JSON with trace_id, PR number, files_checked, findings_count, duration_ms, status
Alert threshold: error_rate > 5% in any 1-hour window → PagerDuty alert to on-call
Incident playbook entry:
  1. Find failing PR's trace_id in the error alert
  2. Filter structured logs by trace_id, sort by timestamp
  3. Identify the tool call with status: "error"
  4. Check args on that entry — which file or API call failed?
  5. Verify GitHub token validity if tool is the GitHub MCP call
  6. Check rate limit headers in the log entry's result object
