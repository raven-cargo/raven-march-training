Section 0 · Concept Introduction

What Is a Skill?

Every new Claude Code session starts with a blank slate. No memory of your codebase conventions, your preferred commit style, your API patterns. You'd have to re-explain everything, every time — unless you use skills.

The Problem Skills Solve

A skill is a markdown file with YAML frontmatter. Claude reads it before performing a specialized task, giving it persistent expertise that survives across sessions. Skills live at:

  • ~/.claude/skills/your-skill.md — global, available in every session
  • .claude/skills/your-skill.md — project-scoped, available only in that project

Anatomy of a Minimal Skill File

A skill has two parts: YAML frontmatter (metadata) and a Markdown body (instructions).

--- ← frontmatter starts name: git-commit-writer description: Writes conventional commits with scope, body, and breaking change notes triggers: - "write a commit" - "commit message" - "conventional commit" version: "1.0.0" --- ← frontmatter ends # Git Commit Writer ← body starts here (plain Markdown) Write commit messages following the Conventional Commits spec. Format: `<type>(<scope>): <subject>` Types: feat, fix, docs, refactor, test, chore Always include: Why this change (not what — the diff shows what). Breaking changes: Add `BREAKING CHANGE:` footer.
CALLBACK — Lab 05, Section 2

TCEF Prompt Pattern — introduced in Lab 05: Prompt Engineering

The skill body IS a TCEF prompt written once and reused infinitely. Task: what Claude should do. Context: conventions to follow. Examples (optional but powerful). Format: output structure. Writing a great skill body is exactly the same skill as writing a great TCEF prompt — the difference is persistence.

Section 2 · Interactive Workshop

Writing the Skill Body

The skill body is instructions. The difference between a good skill and a useless one is specificity — vague instructions produce vague behavior.

Four Elements of a Strong Skill Body

1
Role statement — What Claude is acting as.
Example: "You are a code reviewer specializing in security vulnerabilities in Node.js APIs."
2
Procedure — Step-by-step numbered process to follow.
Numbered steps prevent Claude from skipping phases or reordering actions.
3
Examples — At least one input/output pair for pattern-setting.
One concrete example is worth ten paragraphs of description.
4
Anti-patterns — Explicit Don't do this sections.
Naming what NOT to do is often more reliable than describing what to do.

Skill Body Grader

Rate each snippet on the 4-level scale, then submit to see the actual grade and reasoning.

Snippet 1

Help the user with their code.

Snippet 2

Review code for security issues. Check for SQL injection, XSS, and auth problems. Report findings.

Snippet 3

You are a security-focused code reviewer for Node.js applications. Procedure: 1. Read the entire file before commenting 2. Identify injection vulnerabilities (SQL, command, path traversal) 3. Check authentication and authorization boundaries 4. Assess secret exposure (hardcoded keys, env var leakage) 5. Rate each finding: Critical / High / Medium / Low Output format: ## Findings - [SEVERITY] Finding description - File: path, Line: N - Fix: specific remediation Never suggest "security through obscurity" as a fix. Never mark a finding as Low if it exposes user data.

Challenge CE-06-C: Author a Full Skill From Scratch

Write a complete, production-quality skill YAML for a standup-writer skill. This skill reads recent git commits (last 24 hours), filters to the current user's commits, groups by type (feat/fix/chore/test), and outputs: "Yesterday: X. Today: Y. Blockers: Z."

Required fields — all must be present and substantive:
  • name: snake_case, descriptive
  • description: 2+ sentences — what it reads, what it produces, when to use it
  • triggers: array with at least 4 phrases (natural language, no collisions)
  • behavior: step-by-step instructions referencing the git log format
  • output_format: exact markdown structure for the standup
  • constraints: what NOT to include (merge commits, WIP commits, other authors)
Reference — example of a complete skill YAML (commit-summary):

Study this format — then write yours below. Your standup-writer skill should have the same completeness.

---
name: commit-summary
description: >
  Generates a concise summary of recent git commits for a given time range.
  Reads the git log and formats output as a bullet list grouped by conventional
  commit type. Use when you need a changelog entry or a quick release notes draft.
triggers:
  - summarize my commits
  - what did I commit recently
  - generate release notes
  - create a changelog entry
behavior: |
  1. Run: git log --since=[range] --oneline --no-merges
  2. Parse commit messages for conventional prefixes: feat, fix, chore, docs, test
  3. Group commits by type — most impactful first (feat > fix > docs > chore)
  4. Format each as a bullet with the message and short hash
  5. Do NOT include merge commits or commits with empty messages
  6. Do NOT fabricate commits not in the actual log
output_format: |
  ## Commits — [DATE RANGE]
  **Features:** [feat: items as bullets]
  **Fixes:** [fix: items as bullets]
  **Other:** [remaining items]
constraints: |
  - Only include commits in the specified range
  - Do NOT modify any files
  - Do NOT guess at commit intent from code — use only the commit message
Success criteria (self-check after reveal):
  • Description mentions: git commits, 24-hour window, standup format
  • At least 4 trigger phrases, none overlap with commit-writer or changelog-generator
  • Behavior explicitly calls out git log filtering by author
  • Output format shows Yesterday / Today / Blockers structure
  • Constraints section has at least 3 "do NOT include" items
Model Answer: 
---
name: standup-writer
description: >
  Generates a formatted daily standup from your recent git history.
  Reads commits from the last 24 hours authored by the current git user,
  groups them by conventional commit type, and outputs a structured
  Yesterday / Today / Blockers summary. Use when preparing a standup
  update or before a team sync.
triggers:
  - write my standup
  - generate standup update
  - what did I work on yesterday
  - prepare daily standup
  - summarize my commits
behavior: |
  1. Run: git log --since="24 hours ago" --author="$(git config user.email)" --oneline --no-merges
  2. Parse each commit message for conventional commit prefix (feat:, fix:, chore:, test:, docs:)
  3. Filter out: merge commits (Merge branch...), WIP commits (wip:, WIP), commits by other authors
  4. Group remaining commits by type
  5. For Yesterday: list completed work (feat, fix, docs commits)
  6. For Today: if calendar date has changed, list today's commits; otherwise state "continuing work from above"
  7. For Blockers: scan commit messages for TODO, BLOCKED, or question marks; if none, output "None"
  8. Format and output using the output_format template
output_format: |
  **Daily Standup — [DATE]**

  **Yesterday:**
  - [type] [description] (from commit: [hash])

  **Today:**
  - [planned work based on in-progress items]

  **Blockers:**
  - [None | specific blocker]
constraints: |
  - Do NOT include merge commits or automated commits
  - Do NOT include commits authored by other team members
  - Do NOT include WIP commits (prefix wip: or message contains [WIP])
  - Do NOT fabricate or summarize work not in the commit log
  - Do NOT list commits older than 24 hours unless no recent commits exist

Skills vs. Commands — Side by Side

SkillsCommands
File location ~/.claude/skills/ or .claude/skills/ ~/.claude/commands/ or .claude/commands/
File format YAML frontmatter + Markdown instructions Plain Markdown — no frontmatter required
Invocation Via Skill tool or orchestrator in instructions Via /command-name typed in Claude Code
Purpose Reusable expertise Claude loads when directed User-facing shortcuts that invoke skills or run tasks

A Minimal Command File

# Review PR # File: ~/.claude/commands/pr-review.md Use the Skill tool to invoke the `pr-reviewer` skill. Then apply the review to the current git branch's changes. Arguments: $ARGUMENTS (optional PR number or branch name)
CRITICAL — Reserved Names

NEVER name a command using a Claude Code built-in name. Commands in ~/.claude/commands/ take precedence over built-in commands. A file named plugin.md will shadow the native plugin system — breaking marketplace access and plugin installation entirely.

Reserved NameNative FunctionUse Instead
pluginOfficial plugin marketplaceplugin-builder, my-plugin
helpBuilt-in help systemmy-help, guide
clearClear screencleanup, reset
configConfiguration managementmy-config, settings
mcpMCP server managementmcp-helper, mcp-tools

Challenge CE-06-D: Build a /pr-ready Slash Command

Design a complete /pr-ready slash command that checks whether a branch is ready to submit as a PR. The command must: (1) read the current diff vs. main, (2) check which test files exist for changed source files, (3) read CLAUDE.md for shipping constraints, (4) output a structured readiness report with a go/no-go recommendation.

Fill in each section of the command file:
Success criteria:
  • Arguments section explicitly states whether arguments are required vs. optional
  • Steps reference actual tool calls (git diff, Glob, Read) not just "check the diff"
  • Data flows between steps (step 2 uses output of step 1)
  • Output format has a clear GO/NOT READY verdict with specific evidence
  • CLAUDE.md missing case is explicitly handled (not ignored)
Model Answer:

Arguments: Optional: --base <branch> (default: main). No required arguments — the command should work with zero arguments for the common case.

Steps:

  1. Run Bash: git diff --name-only origin/main...HEAD → collect list of changed file paths
  2. For each changed file matching src/**/*.{ts,js}, run Glob to check if **/*.test.{ts,js} exists for the same module. Record: file name, has_test (true/false)
  3. Run Read: CLAUDE.md. If file not found: set constraints = "No CLAUDE.md found — no documented constraints. Proceed with standard review." If found: extract any lines under "## Shipping" or "## Deploy" or "## Constraints"
  4. Output the structured report using all collected data. GO if: at least 80% of src files have tests AND no CLAUDE.md constraint is violated. NOT READY otherwise with specific reason.

Output format:

## PR Readiness Report
**Branch**: [branch-name] vs [base]
**Changed Files**: N files (M in src/, K test files)

**Test Coverage**:
| File | Has Test? |
|------|-----------|
| src/auth.ts |  Yes |
| src/utils.ts |  Missing |

**CLAUDE.md Constraints**:
- [constraint 1]
- No CLAUDE.md found (if applicable)

**Recommendation**:  READY TO SUBMIT /  NOT READY
Reason: [specific reason if not ready]

Section 4 · Concept Introduction + Simulation

Skill Composition

Individual skills are powerful. Composed skills — skills that invoke other skills — are how you build complex agentic workflows that run reliably without manual coordination.

Three Composition Patterns

1 · Sequential

Skill A → Skill B → Skill C — a pipeline where output of each feeds the next

invoke `pr-reviewer` → if passes → invoke `git-commit-writer` → invoke `changelog-updater`
2 · Conditional

If [condition] → invoke Skill A, else → invoke Skill B

if review_passed: invoke `git-commit-writer` else: invoke `fix-suggester` with findings as context
3 · Parallel

Launch Skill A and Skill B simultaneously via the Agent tool in the skill body

Agent(task="run `security-scanner`") + Agent(task="run `test-coverage-checker`")
CALLBACK — Lab 01, Section 1

PRAO Loop — first introduced in Lab 01: The Paradigm Shift

Skill composition maps directly onto the PRAO loop. Each skill invocation is an Act. Its output is immediately available as input to the next skill's Perceive phase. A composed skill pipeline is a PRAO loop running multiple times in sequence, each cycle narrowing in on a complete result.

Simulation: The ship-feature Skill Executing

Step through the execution to see which skill runs at each stage, what output passes forward, and where human approval gates exist.

INVOKE
ship-feature skill starts. Reads the current branch diff and passes it as context to the next skill.
↓
INVOKE → pr-reviewer
Receives the diff. Runs the 5-step review procedure. Output: findings list with severities.
↓
⏸ HUMAN GATE
Review findings displayed. User approves or requests fixes. Pipeline pauses here — no action taken without human sign-off.

Model Answer

1. Four skills for a FastAPI project:

  • api-documenter — Generates OpenAPI 3.0 spec YAML from FastAPI endpoint definitions
  • test-writer — Writes pytest tests with fixtures and async client setup for new endpoints
  • migration-generator — Creates Alembic migration files from SQLAlchemy model changes
  • code-reviewer — Reviews Python code for FastAPI patterns, Pydantic v2 correctness, and async safety

2. Complete skill file — api-documenter:

--- name: api-documenter description: Generates OpenAPI 3.0 specification YAML from FastAPI endpoint source code triggers: - "document the API" - "write OpenAPI spec" - "generate API docs" - "document this endpoint" version: "1.0.0" --- # API Documenter You are an OpenAPI 3.0 specification writer for FastAPI applications. ## Procedure 1. Read the target endpoint file(s) — identify all route decorators (@app.get, @app.post, etc.) 2. Extract path parameters, query parameters, and request body schemas from type annotations 3. Identify response models (Pydantic schemas) and map them to OpenAPI response objects 4. Generate a complete OpenAPI 3.0 YAML block for each endpoint 5. Validate that all $ref references resolve to defined schemas ## Output Format ```yaml /endpoint-path: method: summary: One-line description parameters: [...] requestBody: {...} responses: 200: description: Success content: application/json: schema: $ref: '#/components/schemas/ResponseModel' ``` ## Anti-patterns Never invent fields that are not present in the source code Never omit error responses (400, 422, 500 are standard for FastAPI) Never use OpenAPI 2.0 (Swagger) format — always OpenAPI 3.0

3. /ship command file:

# Ship # File: .claude/commands/ship.md Use the Skill tool to invoke the `code-reviewer` skill on the current branch's changes. If the review passes (no Critical or High severity findings): Use the Skill tool to invoke the `api-documenter` skill on any modified endpoint files. Then write a conventional commit message summarizing the changes. If the review finds Critical or High issues: Stop and report the findings. Do not commit. Arguments: $ARGUMENTS (optional — specific files to scope the review)

4. Composition anti-pattern to avoid:

Anti-pattern: Skills that are too broad ("do everything")

A skill called full-review-and-document-and-commit that tries to do code review, documentation, and commit writing in one body becomes unreliable and impossible to test. When it fails, you can't tell which responsibility broke. One skill, one responsibility — compose them in the command layer where sequencing is explicit and each step is verifiable.
