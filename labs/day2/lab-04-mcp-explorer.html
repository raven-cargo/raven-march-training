The Extension Problem
Why Claude Code needs MCP

Claude Code ships with 9 built-in tools: Read, Write, Bash, Glob, Grep, and a few others. Powerful — but what if your workflow needs to query your database, search your Jira board, or call your internal API? That's the extension problem. MCP solves it.

Callback · Lab 01 S1 — PRAO Loop
Earlier in Lab 01, you mapped the PRAO loop. Every MCP tool call IS the Act phase — the tool result feeds directly into Observe. We'll see exactly how this maps in Section 4.
History: MCP (Model Context Protocol) was open-sourced by Anthropic in November 2024. It's now supported by dozens of clients and hundreds of community servers — an emerging standard for AI extensibility.

The Three Primitives
Tools · Resources · Prompts — the complete MCP surface

Predict First

Before we reveal the MCP primitives, predict: what categories of functionality would you need to expose to an AI assistant?

"Tools are verbs, Resources are nouns, Prompts are templates." That's the entire MCP surface in one sentence.
Tools Callable Functions
Direction: Claude → MCP Server (Claude calls the tool)
search_issues(query: string, status?: string)
Use when: You want Claude to take an action or retrieve computed data — anything with side effects or real-time computation.
Resources URI-Addressable Data
Direction: MCP Server exposes → Claude reads
confluence://pages/12345 or file:///path/to/doc
Use when: You want to make data available for Claude to read, without the overhead of a function call. Think documents, configs, schemas.
Prompts Reusable Templates
Direction: MCP Server defines → Claude (or user) invokes
review-pr template: structures a code review request
Use when: You want to package common interaction patterns — pre-written instruction sets that shape how Claude approaches a task.

S2
Connecting MCP Servers
Transport types: stdio vs Streamable HTTP

Predict First

How do you think Claude Code knows where to find an MCP server?

Your prediction:

Claude Code reads an mcpServers block — either in your project's .claude/settings.json or your global ~/.claude/settings.json. Two transport types exist:

stdio — Local Subprocess
{ "mcpServers": { "my-db-server": { "command": "node", "args": [ "./mcp-servers/db-server.js" ], "env": { "DB_URL": "${DB_URL}" } } } }

Claude Code spawns this as a child process. Perfect for local tools with direct filesystem access.

Streamable HTTP — Remote Server
{ "mcpServers": { "github-server": { "url": "https://mcp.github.com/api", "headers": { "Authorization": "Bearer ${GITHUB_TOKEN}" } } } }

Claude Code connects over HTTP. Perfect for shared team servers or cloud-hosted services.

SSE Deprecated: SSE (Server-Sent Events) transport was the original remote option. It was deprecated in MCP spec 2025-03-26. New servers should use Streamable HTTP. If you see "transport": "sse" in tutorials, they're outdated.

Apply Task

Write the mcpServers config to connect to a local Python MCP server at ./servers/linear.py that needs a LINEAR_API_KEY environment variable.

Model Answer
{ "mcpServers": { "linear": { "command": "python3", "args": ["./servers/linear.py"], "env": { "LINEAR_API_KEY": "${LINEAR_API_KEY}" } } } }

Key points: command = python3, args array with file path, env with ${VAR} substitution syntax.

S3
Reading Tool Schemas
How Claude understands what tools do and when to call them

Claude reads tool JSON schemas to understand what each tool does and how to call it. The description field is what Claude actually uses to decide when to call a tool — write it for Claude, not for humans.

Interactive Schema Explorer

Click any line to reveal an annotation explaining what Claude does with that field.

// search_issues tool schema
"name": "search_issues", click ▾
name — The tool identifier Claude uses in its tool_call JSON when it decides to invoke this tool. Must be unique within a server.
"description": "Search Linear issues by text query with optional filters", click ▾
description — This is what Claude reads to decide WHEN to use this tool. Claude's reasoning is: "Does my current goal match this description?" Write it for Claude's reasoning, not for human documentation.
"inputSchema": { click ▾
inputSchema — Standard JSON Schema. Claude validates its arguments against this before making the call. If Claude tries to pass an invalid argument, the schema prevents the bad call.
"type": "object",
"properties": {
"query": {
"type": "string",
"description": "Full-text search query"
},
"status": {
"type": "string",
"enum": ["Todo", "In Progress", "Done", "Cancelled"], click ▾
enum — Constrains Claude to only pass valid values. Without this, Claude might pass "in-progress" or "completed" which would fail the API call. Enums prevent invalid calls before they happen.
"description": "Filter by issue status"
},
"assignee": {
"type": "string",
"description": "Filter by assignee username"
},
"limit": {
"type": "integer",
"minimum": 1, "maximum": 100,
"default": 20, click ▾
default: 20 — Claude uses this when the user doesn't specify a limit. Instead of guessing or asking, Claude fills in the sensible default. Makes the tool more autonomous.
"description": "Maximum results to return"
}
},
"required": ["query"] click ▾
required: ["query"] — Only query is required. Everything else is optional — Claude can call the tool with just a search query and omit filters if the user didn't specify them.

Apply Challenge — Write a Tool Schema

Write a JSON tool schema for a create_branch tool that:

  • Takes a required repo (string)
  • Takes a required branch_name (string)
  • Takes an optional from_branch (string, default: 'main')
  • Includes a description that tells Claude when to use it
Model Answer
{ "name": "create_branch", "description": "Create a new git branch in a GitHub repository", "inputSchema": { "type": "object", "properties": { "repo": { "type": "string", "description": "Repository name (owner/repo)" }, "branch_name": { "type": "string", "description": "Name for the new branch" }, "from_branch": { "type": "string", "default": "main", "description": "Source branch to branch from" } }, "required": ["repo", "branch_name"] } }

Note: from_branch is not in required — the default handles it. The description tells Claude this is a branch-creation action, not a listing or reading tool.

Challenge CE-04-B: Design a Production Tool Schema

The apply task had you write a basic schema. This challenge is harder: you're building an MCP server for a project management tool, and you need a tool that lets Claude Code search for issues, filter by assignee and status, and return results in a structured format. Write the complete JSON tool schema.

Tool requirements — the schema must support:
  • Search query string (required)
  • Filter by assignee email (optional)
  • Filter by status (optional — must be an enum: "open", "in_progress", "closed")
  • Limit number of results (optional, with a sensible max)
  • All properties need type and description fields
  • Name must follow snake_case convention
Auto-checker verifies:
  • Name is snake_case (no spaces, no camelCase)
  • Description is a complete sentence (>20 chars)
  • At least 4 properties defined (query + 3 filters)
  • Status property uses "enum" type
  • At least one property in "required" array
  • All properties have both type and description
Model Answer: 
{
  "name": "search_issues",
  "description": "Search project management issues with optional filters for assignee and status, returning a structured list of matching issues.",
  "inputSchema": {
    "type": "object",
    "properties": {
      "query": {
        "type": "string",
        "description": "Full-text search query to match against issue titles and descriptions."
      },
      "assignee": {
        "type": "string",
        "description": "Filter results to issues assigned to this email address. Omit to return issues for all assignees."
      },
      "status": {
        "type": "string",
        "enum": ["open", "in_progress", "closed"],
        "description": "Filter results by issue status. Omit to return issues of all statuses."
      },
      "limit": {
        "type": "integer",
        "description": "Maximum number of results to return. Defaults to 20, maximum 100.",
        "minimum": 1,
        "maximum": 100,
        "default": 20
      }
    },
    "required": ["query"]
  }
}
Key design decisions explained:
  • Only query is required — all filters are optional to maximize flexibility
  • Status uses "enum" to enforce valid values — prevents Claude from sending "active" or "done"
  • Limit has minimum, maximum, and default — protects the server from unbounded queries
  • Assignee is a string (not an ID) — strings are easier for Claude to infer from context

S4
MCP in the PRAO Loop
How MCP tool calls fit into the agentic cycle
Callback · Lab 01 S1 — PRAO Loop
From Lab 01: the PRAO loop — Perceive → Reason → Act → Observe. MCP operations map exactly onto this loop. Click each step below to see how.

Interactive PRAO Trace — MCP Tool Call

Click each step to expand the detail. Each step maps to a PRAO phase.

P
Step 1 — Perceive
Claude reads your mcpServers config and the tool schemas from each server at session start. It now knows: "I have a get_overdue_tasks tool that can query issues by assignee."
R
Step 2 — Reason
Claude decides: "The user wants overdue tasks. I have a get_overdue_tasks tool. Its description matches the goal. The user mentioned 'alice' — that maps to the assignee parameter. I should call it."
A
Step 3 — Act
Claude sends the tool call: { "tool": "get_overdue_tasks", "args": { "assignee": "alice", "limit": 10 } }. The MCP server receives this and executes the database query.
O
Step 4 — Observe
MCP server executes the query and returns: [{id: 'LIN-42', title: 'Fix auth bug', due: '2025-03-01'}, ...]. Claude receives this result in its context window as the observation.
R
Step 5 — Reason Again
Claude incorporates the result into its response: "Alice has 3 overdue tasks. LIN-42 (Fix auth bug) is most urgent at 14 days overdue. Shall I open it in Linear?" The loop completes — or begins again if another action is needed.
This is why MCP matters for agentic systems. It's not just "Claude calling APIs" — it's Claude autonomously deciding WHICH tools to call, WHEN, based on what it's trying to accomplish. The schema is the interface; the PRAO loop is the engine.

Capstone Challenge
Design a complete MCP architecture — 100 XP

Scenario

Your team wants Claude Code to help engineers with three daily tasks: querying the production database for debugging, posting updates to Slack, and managing GitHub issues. Design the complete MCP architecture.

1. How many MCP servers will you create? Why?
Hint: think about separation of concerns and least privilege.
2. List the Tools, Resources, and Prompts for each server
Be specific — name the tools and their purpose.
3. Write the complete mcpServers config block for all servers
Use proper JSON with env var substitution.
4. How will you pass credentials securely?
e.g., DB_URL, SLACK_TOKEN, GITHUB_TOKEN
5. What happens if the database server goes offline while Claude is mid-task?
Think about failure modes in an agentic loop.
Reference Architecture

Servers: 3 (db-server · slack-server · github-server)

Separation of concerns — Each server has minimal permissions. A vulnerability in the Slack server cannot access the database. Servers can be versioned and deployed independently.

  • db-server — Tools: run_query, get_schema, explain_query; Resources: db://tables/{name}
  • slack-server — Tools: post_message, list_channels; Resources: slack://channels
  • github-server — Tools: create_issue, search_issues, add_comment; Resources: github://repos/{owner}/{repo}

Credentials: All via ${VAR_NAME} env substitution. Real values in shell environment or .env — never hardcoded in settings.json.

DB offline: Claude receives an error in the Observe phase. It attempts to continue the task using remaining servers, surfaces the error to the user with context, and does not silently fail or retry indefinitely.
