Agent Thinking with OpenCode

You run a task. Claude responds. But what happened in between? Most engineers treat this as a black box. Today you open it.

After this lab you will be able to:
  • Read a chain-of-thought trace and identify the reasoning step that produced a specific decision

Section 0

The Black Box Problem

Every agent response has three layers. Most engineers only ever see the last one.

1
Thinking
Internal scratchpad. What the agent considers — hypotheses, constraints, trade-offs. Visible only in extended thinking mode.
2
Tool Calls
Concrete actions — Read, Write, Bash, Search. These are the Act phase made visible. Each call has inputs and outputs.
3
Response
Synthesis. What the agent reports back. This is the only layer most people see — without context, it looks like magic.

The same task, two views:

Black Box View
INPUT: "Refactor the auth module" OUTPUT: "I've refactored src/auth/index.ts — separated concerns into AuthService, TokenManager, and UserValidator."

Why didn't it rename the exported class? No idea.

Transparent Trace
THINK: Let me understand the current structure first... TOOL: Read("src/auth/index.ts") THINK: Tightly coupled. Separate concerns. But — renaming the export would be a breaking change. TOOL: Write("src/auth/index.ts", refactored) RESPONSE: Refactored. Public API unchanged.
CALLBACK

PRAO Loop — first introduced in Lab 01, Section 1

Each of the three layers maps to a PRAO phase: Thinking = Reason, Tool Calls = Act, Response = Observe reporting back. The black box problem is a transparency problem — you were watching Act and Response but couldn't see Reason.

Section 2

Tool Call Sequences

Tool calls are the Act phase made visible. A sequence of tool calls is an execution plan. You can read it like a recipe — click each pattern to expand its signature.

1. Read → Think → Write ✓ Healthy
Read() → Think → Write()
When: Single-file change where the agent needs to understand before modifying.
✓ Good: One read, one focused write. Clean and purposeful.
Red flag: If you see Read → Write with no Think, the agent may be acting without reasoning.
2. Read × N → Think → Write × N Watch scope
Read() → Read() → Read() → Think → Write() → Write()
When: Multi-file refactor — agent gathers context across multiple files before making coordinated changes.
✓ Good: Reads cluster before writes. Agent built a complete picture first.
Red flag: Writes appearing before all reads are done — agent is acting on incomplete context.
3. Bash → Bash → Bash May signal confusion
Bash() → Bash() → Bash() → ...
When: Iterative shell exploration — agent is searching for something or trying different approaches.
✓ Good: Each Bash call builds on the last with a clear hypothesis (grep → find → cat pattern).
Red flag: Same or similar Bash commands repeating with no Write afterward — agent is stuck exploring.
4. Read → Write → Read ✓ Observe working
Read() → Write() → Read()
When: Act then Observe — the agent is verifying that its write had the intended effect.
✓ Good: The post-write Read is the Observe phase completing the PRAO cycle. This is healthy self-correction.
Red flag: There is no red flag here — this is the pattern working correctly.
5. Think → Think → Think → (no tool call) ✗ Analysis paralysis
Think → Think → Think → [stalled]
When: Agent is in Reason but cannot commit to an Act — usually because the goal is ambiguous or scope is too wide.
Intervention: Add a concrete starting point. "Start by reading src/auth/index.ts" breaks the paralysis.
✗ If left alone: Agent may produce a vague response that summarizes its uncertainty rather than taking action.

Section 3

When Agents Ask for Help

A clarification request is not a failure. It's the agent being honest about uncertainty rather than guessing. Train yourself to celebrate it — an agent that guesses under ambiguity is far more dangerous than one that asks.

Three types of clarification requests:

1. Scope Clarification

"Should I apply this change to all files or just the one you mentioned?"

Signal: Your prompt had ambiguous scope. The agent found multiple valid interpretations.

Response: Narrow the scope explicitly. "Just src/routes/users.ts"

2. Authority Clarification

"This will delete 847 lines. Should I proceed?"

Signal: Agent correctly checking before an irreversible action. This is safe behavior, not weakness.

Response: Answer yes/no. Don't provide new context — just confirm or abort.

3. Context Clarification

"I found two functions named processPayment. Which should I modify?"

Signal: Multiple matches — agent needs disambiguation. The prompt lacked specificity.

Response: Provide file path + function signature. Then add it to CLAUDE.md.

Intervention Trainer — categorize each agent response:

Which database should I connect to — the one in .env.local or .env.production?
Context — this is a context clarification. The agent found two valid config files and can't know which environment applies. Provide the context: "Use .env.local for this run." Then add the rule to CLAUDE.md.
Should I also update the test files?
Answer the question — this is a scope clarification with a yes/no answer. The agent found scope ambiguity and is asking for authority. Say yes or no. Don't add new context — just confirm the scope.
I found 14 files with TODO comments. Should I fix all of them?
Narrow the scope — 14 files is too broad. The agent is surfacing scope explosion. Pick the 2-3 most critical TODOs and narrow: "Fix only the TODOs in src/api/ — skip tests and docs."
This change will modify the public API signature. Is that intended?
Answer the question — this is an authority clarification about an irreversible action. The agent is correctly checking before a breaking change. Answer yes or no. This is the agent being safe, not confused.
Which error handling strategy do you prefer — return null or throw?
Provide context — and then add it to CLAUDE.md. This is a preference question that will recur across sessions. "We throw typed errors: throw new AuthError('message'). Add this to CLAUDE.md so future sessions don't need to ask."

Section 4

Extended Thinking Mode

Extended thinking is a model-level API parameter available in Claude 3.7 Sonnet. It gives the model a scratchpad it works through before responding — and makes that scratchpad visible to you. This is where the agent reasons through complex problems explicitly.

Use case Extended thinking? Why
Complex architecture decisions ✓ Yes Agent explores trade-offs explicitly — you can audit the reasoning
Debugging multi-file failures ✓ Yes Trace shows which hypothesis the agent tested and why it chose the fix it did
Simple file edits ✗ No Overhead with no benefit — standard mode is faster and equally effective
Boilerplate generation ✗ No No real reasoning required — mechanical task, no trade-offs to surface

An annotated extended thinking excerpt — notice how the agent surfaces an ambiguity it resolved:

thinking
The user wants to migrate from REST to GraphQL. Let me consider the implications:

1. All existing clients use REST endpoints — breaking change for external consumers.
2. Internal services might be easier to migrate — check what uses the API.
3. Could run both in parallel during transition (Apollo Federation approach).
4. The user said 'migrate' not 'add alongside' — but parallel is safer.

Recommendation: Propose parallel running first, ask if that's acceptable.
APPLY
40 characters to submit
CALLBACK

GCCF Pattern — first introduced in Lab 02, Section 1

The agent's thinking trace revealed an ambiguity in the Goal component. A more precise GCCF prompt — "migrate AND deprecate REST by Q3" vs "add GraphQL alongside REST" — would have eliminated the uncertainty before thinking began. The trace shows you where your prompt fell short.

INSIGHT

Extended thinking makes implicit reasoning explicit. The agent would have made the same internal decision without extended thinking — you just wouldn't have seen it. Extended thinking is an observation tool, not a reasoning enhancer. Use it when you need to audit decisions, not when you want better decisions.

BUG REPORT #1247 API returns 500 on user login after recent deploy

Description: Users are getting 500 Internal Server Error when trying to log in. This started after yesterday's deploy (commit a3f2d8c). Auth worked fine before the deploy.

Steps to reproduce:

  1. POST /api/auth/login with valid credentials
  2. Receive 500 error instead of 200 + JWT token

Environment: Production. Node.js API. Auth handled in src/api/routes/auth.ts. Database is PostgreSQL via Prisma.

Relevant code changed in a3f2d8c: "Refactor user model to add profile fields"

auth.ts route (broken, as of a3f2d8c):

// src/api/routes/auth.ts (simplified)
router.post('/login', async (req, res) => {
  const { email, password } = req.body;
  const user = await prisma.user.findUnique({
    where: { email },
    select: { id: true, email: true, passwordHash: true, profile: true }
  });
  if (!user) return res.status(401).json({ error: 'Invalid credentials' });
  const valid = await bcrypt.compare(password, user.passwordHash);
  if (!valid) return res.status(401).json({ error: 'Invalid credentials' });
  const token = jwt.sign({ userId: user.id }, process.env.JWT_SECRET);
  res.json({ token, user: { id: user.id, email: user.email } });
});

Prisma schema diff (from a3f2d8c):

- model User {
-   id           String @id @default(uuid())
-   email        String @unique
-   passwordHash String
- }
+ model User {
+   id           String  @id @default(uuid())
+   email        String  @unique
+   passwordHash String
+   profile      Profile @relation(fields: [profileId], references: [id])
+   profileId    String  @unique
+ }

Error in production logs: PrismaClientKnownRequestError: Unknown field `profile` for select statement on model User

Write exact tool calls: which tool, which arguments, in which order. Don't start with "read everything" — be surgical.

For each tool call, identify its PRAO phase and explain why it belongs there.

Based on the issue description: "Refactor user model to add profile fields" caused 500 on login. What's the most likely root cause?

Write the GCCF-structured prompt you'd give Claude Code to implement the fix.

After Claude fixes the bug, what should the PR description say? Include: what changed, why, how to verify.

What you're demonstrating:
  • Tool call precision — not "read everything," but the right 3 files in the right order
  • PRAO awareness — knowing which phase each tool call serves
  • Root cause reasoning — forming a hypothesis from limited information
  • GCCF prompting — structured fix instructions that don't leave room for the agent to wander
  • PR communication — translating agent work into human-readable context
Model Answer:
Tool calls:
  1. Bash('git show a3f2d8c --stat') — reveals exactly which files changed in the breaking commit
  2. Bash('git show a3f2d8c -- prisma/schema.prisma') — shows the Prisma schema diff
  3. Read('src/api/routes/auth.ts') — read the auth route to see how it queries the user
Root cause:

Most likely: Prisma schema was updated (added non-nullable profile fields) but either (a) the database migration wasn't applied to production, or (b) npx prisma generate wasn't run, so the Prisma client's TypeScript types are out of sync with the schema. The auth route fails when it tries to select or insert a user with the new fields.

GCCF fix prompt:

"Goal: Fix the 500 on POST /api/auth/login by resolving the Prisma schema mismatch from commit a3f2d8c. Context: The commit added profile fields. Prod DB may not have the migration applied. Prisma client may not be regenerated. Constraints: Do not change auth logic — only fix data layer. Show migration status first before applying anything. Format: Explain what you'll check, what you find, and what you'll change before you change it."

Diagnose a Real Trace

Here's a real agent trace. It shows something going wrong. Your job is to diagnose it and write the intervention.

THINK I need to add authentication to the API endpoints
TOOL Read("src/routes/users.ts")
RESULT Found 4 endpoints: GET /users, POST /users, GET /users/:id, DELETE /users/:id
THINK I'll add JWT authentication middleware to all endpoints
TOOL Write("src/routes/users.ts", [adds requireAuth() to all 4 endpoints])
THINK Done. Let me verify by reading the file
TOOL Read("src/routes/users.ts")
RESULT File updated. All 4 endpoints now require authentication.
THINK Wait — POST /users is the registration endpoint. It shouldn't require auth.
TOOL Write("src/routes/users.ts", [removes requireAuth() from POST /users])
THINK Actually, what about GET /users — should that be public too? The prompt said 'add authentication' but didn't specify which endpoints
TOOL Read("src/routes/users.ts")
THINK I'm unsure about GET /users. Let me check if there's any documentation...
TOOL Read("README.md") ...
BOSS CHALLENGE

Write your complete diagnosis and intervention. Address all four parts:

  1. At what point did the agent's uncertainty become visible?
  2. What was missing from the original prompt that caused this?
  3. Write the improved GCCF prompt that would have prevented this.
  4. What CLAUDE.md entry would document this API's auth rules for future sessions?
80 characters to submit

Model Answer

1. When uncertainty became visible

After the first Write — "Wait — POST /users is registration, shouldn't require auth." The agent acted before resolving a constraint it discovered mid-execution. The prompt said "add authentication" without specifying which endpoints are public.

2. What was missing

The Constraints component of the GCCF prompt. No specification of which endpoints should be public vs. protected. "Add authentication" is a goal without constraints — the agent had to guess, and guessed wrong.

3. Improved GCCF Prompt

Goal: In src/routes/users.ts, add requireAuth() middleware to specific endpoints
Context: JWT-based auth; requireAuth middleware already imported
Constraints: POST /users (registration) and GET /users (public directory) must
remain unauthenticated. GET /users/:id and DELETE /users/:id require auth.
Do not modify endpoint logic — only add/remove middleware calls.
Format: Show the modified file with inline comments marking each change

4. CLAUDE.md Entry

## API Authentication Rules — /users routes
Public (no auth): POST /users (registration), GET /users (public directory)
Protected (requireAuth): GET /users/:id, DELETE /users/:id
Middleware: requireAuth() — JWT validation, already imported in routes files

Self-evaluation — how did you do?

Distinguished
All 4 components present. GCCF lists specific endpoint names. CLAUDE.md entry is reusable across future sessions. Uncertainty pinpointed to the exact trace line.
Proficient
3 components present. GCCF goal is measurable. Missing CLAUDE.md entry or GCCF lacks specificity on endpoint names.
Developing
2 components. Diagnosis correct but prompt improvement is vague ("be more specific"). CLAUDE.md entry absent.
Beginning
1 component or responses are generic ("the prompt needed more detail"). No specific GCCF or CLAUDE.md entry written.
