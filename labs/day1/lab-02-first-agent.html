Section 0

Why Prompts Fail

Every bad output traces to a specific failure point in the PRAO loop. Let's find them.

CALLBACK

PRAO Loop — first introduced in Lab 01, Section 1

Perceive → Reason → Act → Observe — the four-phase agentic execution cycle. Every agent response starts with Perceive. If Perceive gathers bad data, Reason works from flawed input, Act executes the wrong plan, and Observe confirms the wrong result.

Notice how it appears below: each bad prompt breaks at a specific PRAO phase.

Three prompts. Three failures. Three different phases.

Example 1 — Perceive Fails
claude -p "fix my code"
↓ Output
Claude edits README.md instead of your broken function
Perceive had no file context. It guessed.
Example 2 — Reason Fails
claude -p "improve this"
↓ Output
Claude renames variables and reformats the file — none of which you needed
Reason had no goal to guide it. "Improve" is undefined.
Example 3 — Both Fail
claude -p "add tests"
↓ Output
Claude writes test functions with zero assertions — they all pass trivially
Perceive: no file. Reason: no definition of "test". Both upstream phases failed.
The root cause is always upstream.

The quality of Perceive determines everything. The agent can only reason about what it gathered. The fix is always upstream — at the point where Perceive received insufficient context. You cannot patch a bad output; you must fix the prompt that created it.

The GCCF Anatomy

G
Goal
What exactly to produce. Not "improve this" — a specific, measurable outcome.
"Add TypeScript return types to all exported functions in src/utils.ts"
C
Context
What the agent needs to know first. File paths, project setup, existing patterns.
"This is a Node.js 20 project using strict TypeScript. The file uses ES modules."
C
Constraints
What NOT to do. Guard rails that prevent over-engineering or scope creep.
"Don't rename variables. Don't change function signatures. Keep changes minimal."
F
Format
How to structure the output. Prevents Claude from choosing an inconvenient form.
"Show the complete modified file, not a diff. Include a summary of changes made."
Why GCCF maps directly to PRAO:

Goal + Context + Constraints all feed Perceive — they define what the agent gathers and what to ignore. Format shapes how Act packages the result. A GCCF prompt doesn't just describe a task — it pre-loads the agent's perception with the right signal.

INSIGHT

Every word either helps or hurts Perceive. GCCF is a filter — Goal tells Claude what, Context tells it where/why, Constraints tell it what not to do, Format tells it how to package the result. Notice how eliminating ambiguous words forces you to be specific. Specificity is what Perceive needs.

Section 3

When Things Go Wrong

Three error signatures. Each has a distinct cause and a specific fix.

Quick Reference — Error Signatures
ErrorSignatureFix
Permission denied Error: Permission denied for tool: Write Add the tool to the allow list in settings.json
No context — confusion Claude asks clarifying questions about the target Add file paths and project context to the prompt (fix the Context in GCCF)
Tool timeout Task times out after 30s Reduce scope; use claude -p for focused subtasks on specific files

Error Recovery Trace

Click each step to see what's happening and why.

TOOL CALL Write("api/routes.py", updated_content)
Act phase: Claude attempts to write the updated file. This is correct behavior — it has a plan and is executing it.
ERROR Permission denied for tool: Write on path api/routes.py
Observe phase: Claude sees the error. The Write tool is not in the allow list in settings.json. This is a configuration issue, not a prompt issue.
RESPONSE I can't write to api/routes.py — the Write tool isn't in your allow list. Add it to settings.json to proceed.
Observe phase continues: Claude reports precisely what failed and how to fix it. This is the agent being helpful — it's surfacing the exact action you need to take.
YOU Added Write to settings.json allow list. Please continue.
Your action: You update settings.json to include "Write" in the allow array, then tell Claude to continue. The conversation context is preserved — Claude doesn't need to re-read the file.
SUCCESS Write("api/routes.py") → 247 lines written. JSDoc added to all 12 functions.
Act completes, Observe confirms: The write succeeded. Claude then verifies: 12 functions targeted, 12 functions updated. The PRAO loop completed successfully after you resolved the configuration issue.

Section 4

Conversation Threading

Claude has no memory between separate invocations. Here's how to work with that — not against it.

Three modes of memory:

Within one claude session: context accumulates line by line. Between sessions: it disappears entirely — unless you write it down somewhere Claude will read next time. That somewhere is CLAUDE.md.

CALLBACK

CLAUDE.md — first introduced in Lab 01, Section 4

Persistent context file — a briefing document you write once, that Claude reads at the start of every session. Here it solves the "explain again" problem: instead of re-explaining your project conventions each run, you write them to CLAUDE.md and they are automatically part of every agent's Perceive phase.

Mode 1 — Stateless

Single-shot with claude -p

# Each invocation is independent — no prior context claude -p "Add error handling to fetchUser in src/api.ts" # Done. Next run starts fresh. claude -p "Add error handling to fetchOrders in src/api.ts" # Does NOT know about the previous run's changes

Use when: focused, self-contained tasks where prior context doesn't matter.

Mode 2 — Accumulating Context

Interactive session

claude # > Add error handling to fetchUser # Claude does it, sees the result # > Now apply the same pattern to fetchOrders # Claude knows what pattern it just used — applies it consistently

Use when: tasks build on each other and you need the agent to maintain consistency within a session.

Mode 3 — Persistent via CLAUDE.md

Cross-session context

# CLAUDE.md contains: # "This codebase uses Result<T,E> for error types — never throws" # "All async functions must have JSDoc with @throws documented" claude -p "Add error handling to fetchUser" # Knows Result<T,E> pattern — uses it without being told # Next week, new session: claude -p "Add error handling to fetchOrders" # Still knows Result<T,E> — CLAUDE.md persists across restarts

Use when: project conventions, architecture decisions, or team patterns need to be consistent across all runs.

10 Minutes Before the Demo

⏱ Scenario

You've just inherited a codebase. A demo is in 10 minutes. The client expects a working async API layer with comprehensive error handling — currently it has none. You have Claude Code. Write your complete interaction plan now.

CAPSTONE CHALLENGE

Write your complete interaction plan

Your response must address all four components:

  1. The exact GCCF prompt you'd send to Claude Code
  2. One CLAUDE.md entry you'd write before running it
  3. One error you'd expect and your response to it
  4. How you'd verify the output is correct before the demo
80 characters to submit

Model Answer

1. GCCF PROMPT: "In src/api/routes.py, wrap all async endpoint functions in try/except blocks returning ApiError(status=500, message=str(e)). Context: FastAPI project, Pydantic v2, all endpoints return ApiResponse[T]. Constraints: don't add logging (handled by middleware), keep existing return types. Format: show each modified function with before/after." 2. CLAUDE.md ENTRY: ## Error Handling Convention Endpoints return ApiResponse[T] | ApiError; never raise; catch at boundary. ApiError signature: ApiError(status: int, message: str) 3. EXPECTED ERROR + RESPONSE: Permission denied for tool: Write on src/api/routes.py → Fix: Add "Write" to allow list in settings.json, then: "Please continue." 4. VERIFICATION: claude -p "List all async functions in src/api/routes.py that still lack try/except blocks" # Expected output: empty list (all functions covered)

Self-Evaluation Rubric

Distinguished
All 4 components present. GCCF is specific and measurable. Error scenario is realistic and recovery is concrete.
Proficient
3 of 4 components present. Goals are measurable. CLAUDE.md entry or verification step may be missing.
Developing
2 components present. GCCF prompt may exist but lacks Constraints or Format.
Beginning
1 component or entirely vague responses. Plan would not be actionable in a real scenario.
