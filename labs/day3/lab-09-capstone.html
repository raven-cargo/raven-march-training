Callback — PRAO Loop (Lab 01)

Your pipeline IS the PRAO loop at scale.

You first met Perceive → Reason → Act → Observe in Lab 01, Section 1. Here it is doing real work: What does Perceive receive? What does Reason decide? What does Act execute? What does Observe confirm? Every pipeline component maps directly to one of these four phases.

Five pipeline components every design must address:

Component 1

Trigger

What event starts the pipeline?

webhook, cron schedule, file change, Slack message, GitHub PR event
Component 2

Context Gathering

What does the agent need to Perceive before reasoning?

files, logs, tickets, schemas, git history, CI results
Component 3

Decision Logic

What does the agent decide or produce?

classification, draft content, structured report, risk score, summary
Component 4

Action Scope

What can the agent Act on — vs. what requires human approval?

post comment (auto), merge PR (human gate), delete file (never)
Component 5

Output Delivery

How does the result reach humans?

PR comment, Slack DM, email, Jira ticket, dashboard widget
Apply — Architecture Design

Map your problem to all five pipeline components. Be specific about each — not "reads relevant files" but "reads the changed files from git diff and the test results from CI logs."

Insight

Specificity transforms a vague idea into a buildable spec.

"Reads relevant files" requires a design decision. "Reads git diff --name-only HEAD~1 and the last 3 lines of each changed file" is buildable today. Notice how each component forced you to make a concrete choice — those choices are your design. They can be reviewed, challenged, and improved. Vague intentions cannot.

CE-09-A: Write the Actual Skill YAML

You've described your skill in the design section above. Now write it. Produce a complete, syntactically valid skill YAML for one skill in your pipeline. Not a design — the actual file you'd save to ~/.claude/skills/.

Required fields (validation will check all of these):
name
snake_case, unique identifier
description
>3 sentences, what+when+why
triggers
≥3 natural language phrases
behavior
>100 words, must include constraints
output_format
exact structure, not "markdown"
examples
≥1 concrete before/after example
Reference format — a complete skill YAML (for format guidance only):

Write your pipeline's skill below — not this example. This shows the expected completeness of all 6 fields.

---
name: pr-description-writer
description: |
  Writes a structured PR description from the current git diff vs main.
  Reads changed files, extracts intent from commit messages and code, and produces
  a description with Summary, Changes, and Testing sections. Use before opening a PR.
triggers:
  - write my PR description
  - generate PR summary
  - draft pull request description
behavior: |
  1. Run git diff --name-only origin/main...HEAD to get changed files
  2. Read each changed file and its diff (Read tool per file)
  3. Read recent commit messages for intent context
  4. Generate the PR description using the output_format template
  5. Do NOT invent test results or claim tests pass if not verified
  6. Do NOT include internal implementation details in the summary
output_format: |
  ## Summary
  [1-2 sentences: what this PR does and why]

  ## Changes
  - [filename]: [what changed and why]

  ## Testing
  - [ ] [how to verify each change manually]
examples: |
  Input: "write my PR description"
  Output: ## Summary\nAdds rate limiting to /api/auth/login to prevent brute-force attacks.\n
  ## Changes\n- src/middleware/ratelimit.ts: new middleware, 100 req/15min per IP\n
  ## Testing\n- [ ] POST /api/auth/login more than 100 times in 15 min — expect 429
Skill YAML validated!

Your skill file is ready to save to ~/.claude/skills/[name].md. This is the actual artifact — not a description of an artifact.

Section 3

Safety & Production Design

Before this pipeline touches production, answer three questions: what can it touch? what can it cost? who reviews before it acts?

Predict Before You Read

How would you structure a CLAUDE.md for a pipeline that runs unsupervised? What sections would you include to keep the agent safe without a human watching every action?

Callback — Permission Scoping (Lab 08, Section 1)

Principle of least privilege applied to agentic pipelines. In Lab 08, you learned to write the settings.json permissions block that controls what tools and file paths an agent can access. That skill is load-bearing here — write the permissions block for your pipeline now.

Three production gates every pipeline needs:

1

Permission Scope

Explicit allow/deny in settings.json. What tools are allowed? What file paths or domains can the agent access? What is explicitly denied? Least privilege means the agent can do exactly what it needs — and nothing more.

2

Human Approval Gate

Which actions require a human to confirm before execution? Any action that is difficult to reverse, has external side effects, or could cause data loss must have a human gate. Identify the single most consequential action in your pipeline and gate it.

3

Failure Behavior

What does the pipeline do when an agent fails or produces unexpected output? Fail silently? Post an error notice to Slack? Alert on-call? Write to a dead-letter queue? An unhandled failure in production is worse than a slow pipeline.

Reference: settings.json structure

// settings.json — Claude Code permissions for your pipeline agent { "permissions": { "allow": [ "Bash(git:*)", // allow all git subcommands "Read(src/**)", // read source files "WebFetch(api.github.com)" // fetch from GitHub API ], "deny": [ "Write(*)", // no file writes "Bash(rm:*)", // no deletions "Bash(git push:*)" // no push operations ] } }

CE-09-C: Issue Tracker Integration Specification

Production pipelines don't run invisibly — they create paper trails. Specify how your pipeline integrates with an issue tracker (Linear or GitHub Issues). This is a design spec — specific enough to implement, not vague enough to ignore.

Be specific: "when the orchestrator dispatches a subtask" not "when something starts"

List the exact fields: title, description, labels, assignee, metadata you'd include

What are the human checkpoints? When does the pipeline pause and wait?

No-response handling: escalate? skip? cancel? retry?

What makes a good integration spec vs. a vague one:
Too vague:
"Create issues when tasks start and close them when done"
Specific enough:
"When orchestrator calls dispatch_task(), call mcp__linear__create_issue() with title='[task_id]: [description]', status='In Progress'"
Integration spec submitted!

This spec is now part of your capstone design. A pipeline with clear tracker integration is a pipeline you can debug, audit, and hand off to teammates.

Peer Review Protocol

The final quality gate: a structured self-review. Score your design across five professional dimensions before you export it.

Rate your own design across five dimensions. Use the behavioral anchors — not instinct.

Select a level for each dimension. Your weighted score will calculate automatically.

1. Correctness

Does the pipeline actually solve the stated problem? Is the trigger → context → decision → action → output chain complete?

25%

2. Architecture

Is the design justified? Are trade-offs explicit? Is the agent pattern (single vs. multi-agent) appropriate for the problem?

25%

3. Safety

Does the permission scope follow least privilege? Is there a human gate on irreversible or consequential actions?

20%

4. Adaptability

Could you swap one component (e.g., GitHub to GitLab) without redesigning the whole pipeline? Is the design modular?

15%

5. Communication

Could another engineer build this from your design document alone, without asking you any questions?
