Permissions — the settings.json system

Learning objective Configure allow/deny permissions so the agent can work autonomously within safe boundaries. Understand how every action resolves through the three-state model: permitted, blocked, or ask.
How every agent action resolves
In allow?
→Execute immediately, no prompt
In deny?
→Blocked entirely, agent cannot proceed
Neither?
→Ask human (interactive) or skip (non-interactive)
Example settings.json
{ "permissions": { "allow": [ "Read", "Write(src/**)", "Write(tests/**)", "Bash(npm test)", "Bash(npm run build)", "Bash(npx tsc *)", "Bash(git status)", "Bash(git diff *)" ], "deny": [ "Bash(rm *)", "Bash(sudo *)", "Bash(git push *)", "Bash(git commit *)", "Write(.env*)", "Write(config/prod*)" ] } }
Test: how does each action resolve?
▶ Click each action to see if it's permitted, blocked, or requires approval — and which rule matched.
Key principles
1
Least privilege. Configure only what the agent needs for the current task. Refactoring src/? It doesn't need Write access to config/.
2
Approval as signal. If the agent keeps asking to approve the same action, either add it to allow (if you trust it) or investigate why it's attempting it.
3
Two layers stack. Global (~/.claude/settings.json) sets your baseline. Project (.claude/settings.json) adds project-specific permissions on top.
