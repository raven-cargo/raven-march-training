MCP servers — extending agent capability

Learning objective Understand MCP as the way to give Claude Code access to external tools (databases, APIs, issue trackers). Know the two transport types, how to configure them, and how to debug when they don't connect.
Without MCP, Claude Code can only read files and run bash commands. With MCP, it can query databases, create issues in Linear, fetch Notion pages, search vector stores — any external tool exposed through the protocol. MCP tools work identically to built-in tools once connected.
▶ Click each transport type to see configuration examples and when to use them.
stdio (local)
Subprocess on your machine
Claude Code starts it, talks via stdin/stdout, kills it when the session ends. Best for local dev.
Streamable HTTP (remote)
External service over network
Persists independently. Used for shared team servers, cloud APIs, SaaS integrations. Replaces older SSE transport.
When it doesn't connect
▶ Walk through these steps in order — they cover the three most common failure points.
1
Test the command manually — does the server process start?
npx -y @modelcontextprotocol/server-postgres "postgresql://localhost/mydb"
2
Check environment variables — are credentials actually set?
echo $LINEAR_API_KEY
3
Start Claude Code and verify tools are available
claude → "What tools are available from the postgres MCP server?"
MCP tools use the same permission model. Allow or deny with mcp__{server}__{tool} in settings.json. Example: allow mcp__postgres__query but deny mcp__postgres__execute for read-only access.
