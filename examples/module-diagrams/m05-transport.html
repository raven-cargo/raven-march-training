Transport configuration

Configuration Two active transports: stdio (local subprocess) and Streamable HTTP (remote service). Know the config format for each, plus the three-phase connection handshake that loads tool schemas into the session.

Communication
HTTP POST requests, supports streaming responses
Lifecycle
Persists independently — always running
Best for
Shared team servers, cloud APIs, SaaS integrations
Config fields
url, headers (for auth via ${VAR})
{ "mcpServers": { "shared-tracker": { "url": "https://mcp.example.com/tracker", "headers": { "Authorization": "Bearer ${MCP_AUTH_TOKEN}" } } } }
CLI shorthand
claude mcp add --transport http shared-tracker https://mcp.example.com/tracker --header "Authorization: Bearer $TOKEN"
SSE (Server-Sent Events) was deprecated in the MCP specification on 2025-03-26. It used GET with SSE for server→client and POST for client→server — a split mechanism superseded by unified Streamable HTTP. If you encounter SSE in existing configs, migrate to Streamable HTTP. Current servers and clients do not support it.
The connection handshake
Three phases when Claude Code connects to an MCP server.
1
Initialize
Client sends protocol version and capabilities. Server responds with its version, capabilities, and server info. Incompatible versions = connection fails here.
initialize → { protocolVersion, capabilities, serverInfo }
2
List capabilities
Client requests all tools, resources, and prompts. Server returns all schemas. Claude loads them into the session's capability inventory.
tools/list → schemas | resources/list → URIs | prompts/list → templates
3
Available
Connection active. Claude can call tools, read resources, and request prompts. Persists until the session ends.
tools/call, resources/read, prompts/get → active for session
Practical implication: MCP server changes (adding tools, modifying schemas) don't take effect until the next session — schemas are loaded at connection time during phase 2.
