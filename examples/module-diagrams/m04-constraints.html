Constraint specification

Safety & precision Constraints define what the agent must not do. Four categories cover the space: scope (what's in/out), compatibility (what must be preserved), style (conventions to follow), and approval (where to pause for human confirmation).

Define what files, functions, and systems are in and out of scope. Prevents the agent from wandering into adjacent code that shouldn't be touched.
Verifiable constraint
Only modify files in src/auth/. Do not modify files in src/middleware/, src/models/, or any test files.
Meaningless constraint
Don't break anything outside the auth module.
Why the bad version fails: "Don't break anything" requires the agent to know all downstream consumers of every file it touches. It can't verify this. Explicit file paths are verifiable.
Define interfaces, behaviors, and formats that must be preserved. Critical for refactoring and migrations where the internal implementation changes but the external contract must not.
Verifiable constraint
The function signature validateToken(token: string): Promise<User | null> must remain unchanged. All 47 existing unit tests must pass without modification.
Meaningless constraint
Make sure it's backward compatible.
Why the bad version fails: "Backward compatible" is undefined without specifying which interfaces, which callers, and which behaviors. The good version names the exact function signature and a measurable test suite.
Define conventions the output must follow. Ensures new code matches existing codebase style — generic correct ≠ project-correct.
Verifiable constraint
Use camelCase for all new variables and functions. Use snake_case for database column names. Use parameterized queries — no string concatenation in SQL.
Meaningless constraint
Follow best practices and keep it clean.
Why the bad version fails: "Best practices" varies by community, language, and era. "Clean" is a subjective aesthetic. The agent can't make a binary decision from either. Specific naming conventions are checkable.
Define points where the agent must pause and get human confirmation before proceeding. The most important safety mechanism in agentic workflows — converts autonomous executor to collaborative planner.
Verifiable constraint
Before writing any code, present a numbered list of changes you plan to make, specifying which files and what modifications. Do not write code until I reply "proceed."
Meaningless constraint
Be careful with your changes.
A well-formed approval constraint specifies three things: what must be shown before proceeding (numbered list of changes), what format the plan takes (files + modifications), and what the agent waits for (the word "proceed").
