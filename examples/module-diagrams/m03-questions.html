Clarifying questions

Mindset shift A clarifying question means the agent encountered genuine ambiguity and is asking instead of guessing. This is calibrated behavior — the alternative is silent wrong assumptions. Learn to recognize the three types and answer them productively.
Agents that don't ask questions either have excellent context (ideal) or make silent assumptions (dangerous). Silent assumptions produce confident wrong code. A clarifying question is a feature, not a failure.

"Should I also update the logout endpoint to invalidate refresh tokens, or is that a separate task?"
Productive answer
"Include logout — invalidate the refresh token in the database when the user logs out. That's part of this task."
Specific scope decision. The agent can now proceed without ambiguity.
Unproductive answer
"Do what makes sense."
Forces the agent to guess at scope. It may include logout (adding scope) or skip it (missing critical work).
Record to CLAUDE.md? No — this is a one-off scope decision for this task. If you find yourself repeatedly answering scope questions about what "auth refactor" includes, then yes.
"Store refresh tokens in PostgreSQL (new table, persistent, queryable) or in-memory store (simpler, faster, but not persistent across restarts)?"
Productive answer
"Use PostgreSQL — we don't have Redis in production, and refresh tokens need to survive restarts. Create a new refresh_tokens table."
Specific decision with reasoning. "Production doesn't have Redis" affects downstream decisions the agent will make.
Unproductive answer
"Whatever you think is best."
This is an architectural decision the agent correctly escalated. "Whatever you think" delegates authority for a choice that affects production infrastructure.
Record to CLAUDE.md? Yes — this is a standing architectural decision. Add under Decided: "Refresh tokens stored in PostgreSQL (refresh_tokens table). No Redis in production." This prevents re-asking in future sessions.
"Should new refresh token tests use the existing JWT mock (fast, isolated) or test with real JWT sign/verify operations (slower, but catches encoding issues)?"
Productive answer
"Use real JWT operations for the refresh token tests — we've had encoding bugs before. Keep the existing tests with mocks as they are."
Specific approach with reasoning. The agent now knows the testing philosophy for this specific area.
Unproductive answer
"Follow the existing test patterns."
The agent is asking precisely because existing patterns don't clearly apply to this new case. Pointing back to ambiguous source doesn't resolve it.
Record to CLAUDE.md? Yes, if it's a standing convention — add under Conventions: "Auth tests use real JWT operations, not mocks. Other tests can use mocks." If one-off, leave in conversation.
