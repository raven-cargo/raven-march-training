The extension problem MCP solves

Why MCP exists Before MCP, every agent integration was bespoke — custom prompts, custom parsing, custom error handling. MCP standardizes the protocol so capabilities are composable, auditable, and portable across any MCP-compatible client.

Every integration is a custom cable — built for one device, unusable with any other. Each new system means starting from scratch: new prompts, new parsing, new error handling.
The USB-C analogy: MCP is to AI capabilities what USB-C is to hardware. One standard protocol — any MCP client connects to any MCP server. Capability author implements the server once; every compatible client gains access.
Discovery
Client learns what server provides
tools/list, resources/list, prompts/list
Invocation
Client calls a capability
tools/call, resources/read, prompts/get
Schema
Capabilities describe themselves
JSON Schema for inputs, URIs for resources
Transport
How client and server communicate
stdio (local), Streamable HTTP (remote)
✓Composable: combine multiple MCP servers in one session — database + issue tracker + docs
✓Auditable: every tool call is a structured request with typed parameters, not free-text HTTP
✓Portable: same MCP server works with Claude Code, any compatible framework, any future client
